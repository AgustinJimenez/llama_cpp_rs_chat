AGENTS GUIDE

This is the short canonical reference for agents (Claude Code, OpenAI Agents, etc.) working in this repo.

What this is: Multi-provider AI chat app (React + Rust). Supports local models (llama.cpp GGUF), CLI-backed providers (Claude Code, Codex), and 13+ OpenAI-compatible cloud providers (Groq, Gemini, SambaNova, Cerebras, Mistral, etc.) with agentic tool loop. Core local flow: load GGUF model → auto-calculate GPU layers from available VRAM → extract Jinja2 chat template + model metadata from GGUF → resolve model-specific tool format and sampler presets → format prompts via Jinja template → generate tokens with configurable sampler chain (11 types) → stream over WebSocket → detect tool call patterns in output (`<tool_call>`, `[TOOL_CALLS]`, `SYSTEM.EXEC`, etc.) → execute native tools (file I/O, shell, web search/fetch, Python) → inject results back into context for multi-turn agentic loops. Cloud providers: OpenAI-compatible `/v1/chat/completions` API with function calling → local tool execution → result injection → loop (max 20 iterations). KV cache reused between conversation turns for speed. Conversations and config persisted in SQLite. Model runs in an out-of-process worker (child process with JSON Lines IPC) — kill worker to reclaim all VRAM instantly. Runs as web app (Vite + Rust backend) or Tauri desktop app.

CUDA/BUILD CRITICAL: Always build with `--features cuda,vision` for GPU inference. Never test model loading with a debug or CPU-only build — the debug binary does not include CUDA and crashes on model load. Debug builds run at ~5 tok/s vs ~80-150 tok/s in release due to unoptimized Rust sampling code. Always verify you are running `target/release/llama_chat_web.exe`. Check with: `wmic process where "name='llama_chat_web.exe'" get ExecutablePath`.

Default development (Tauri commands handle both frontend and backend together): use `npm run dev:auto:desktop` (automatic GPU detection, preferred) or `npm run tauri:dev` (CPU-only desktop). Web app fallback: `npm run dev:auto` (automatic GPU detection) or `npm run dev:cuda` (manual CUDA). CRITICAL: `npm run dev:cpu` is CPU-only — no CUDA, model runs at ~5 tok/s. Always use `dev:cuda` or `dev:auto` for GPU acceleration.

Web app runs Vite on port 14000 with Rust backend on port 18080. Access via http://localhost:14000. Desktop app opens native window.

Web mode vs Tauri mode: In web mode (`llama_chat_web`), the HTTP REST API is available on port 18080. In Tauri mode (`llama_chat_app`), the same HTTP REST API is ALSO served on `127.0.0.1:18080` — the Tauri app spawns it at startup via the shared `web::http_dispatch::serve` (src/main.rs), reusing the desktop DB + the worker it already spawned. Some frontend features (notably the agents UI, `src/contexts/AgentContext.tsx`) call `fetch('/api/...')` with no Tauri-IPC fallback, and Vite proxies `/api → 18080`. Other features (config, model load, chat, conversations) prefer Tauri IPC commands via `tauriCommands.ts`. Both the HTTP API and the IPC commands operate on the same desktop DB. The App-UI MCP server runs on http://127.0.0.1:18091/mcp in Tauri mode for external programmatic interaction. Web browsing tools (`browser_search`, `browser_navigate`) ONLY work in Tauri mode because they use the in-app WebView — they will fail in web-only mode.

IMPORTANT: `cargo tauri dev` builds in DEBUG mode by default — CUDA debug builds take ~27 min. Use `npm run tauri:dev:release` or `cargo tauri dev --release` for faster iteration. Also requires cmake on PATH — if running directly (not via `npm run cargo`), set `CMAKE=target/cmake/cmake-3.31.6-windows-x86_64/bin/cmake.exe`.

GPU acceleration: `npm run dev:auto` (web) or `npm run dev:auto:desktop` automatically detect the best setup (Metal on macOS, CUDA on Windows, CPU fallback). On Windows, `dev:auto` runs `dev:cuda` directly with VS env auto-setup via `ensure-vs.cmd`. Manual options: `npm run dev:metal`/`npm run tauri:dev:metal`, `npm run dev:cuda`/`npm run tauri:dev:cuda`.

Frontend alternatives: `npm run build` for production build. If the backend is already running, start Vite only with `npx vite --host --port 4000`. Backend alternatives: `cargo run --bin llama_chat_web` if Vite is already running.

Desktop app: `cargo tauri dev` for hot reload. Build commands: `npm run tauri:build` (smart — auto-detects GPU), `npm run tauri:build:cuda` (force CUDA), `npm run tauri:build:cpu` (force CPU-only, used for distribution installers). For the distribution installer always use `npm run tauri:build:cpu` — CUDA DLLs must NOT be bundled (user installs CUDA toolkit themselves). The app detects NVIDIA GPU + missing CUDA at runtime and shows an install link. Bundling CUDA bloats the installer from ~20MB to ~654MB. The binary at target/release/llama_chat_app.exe must be the CUDA build (~190MB) for GPU inference — the CPU-only build (~52MB) runs at 15 tok/s. Verify with: `(Get-Item target/release/llama_chat_app.exe).Length / 1MB`.

CMake: required by llama-cpp-sys-2. All npm scripts route through tools/ensure-cmake, a standalone Rust tool that auto-downloads a portable CMake if it's not on PATH. Checks: (1) "cmake" on PATH, (2) well-known install locations, (3) cached download in target/cmake/, (4) fresh download from GitHub releases.

VS developer environment (Windows): `scripts/ensure-vs.cmd` auto-detects Visual Studio via `vswhere`, calls `vcvars64.bat`, and fixes PATH ordering so MSVC's `link.exe` is found before Git for Windows' `link.exe`. It also adds `~/.cargo/bin` to PATH if cargo isn't found. This script is prepended to all build-related npm scripts. CRITICAL: batch files must have CRLF line endings — LF-only causes cmd.exe to fail parsing.

Docker: CPU — `docker build -f Dockerfile.test-cmake -t llama-cpu .` then `docker run -p 8000:8000 -v /path/to/models:/app/models -v ./assets:/app/assets llama-cpu`. CUDA — `docker build -f Dockerfile.cuda -t llama-cuda .` then `docker run --gpus all -p 8000:8000 -v /path/to/models:/app/models -v ./assets:/app/assets llama-cuda`. Mount models to /app/models. CUDA requires NVIDIA Container Toolkit on the host.

Testing: `npm test` (Playwright E2E; backend must be running on 18080). Variants: `npm run test:ui`, `npm run test:headed`, `npm run test:debug`, `npm run test:docker`. Unit tests: `cargo test`. Single browser: `npx playwright test --project=chromium`. Mock mode: build with "mock" feature — `cargo build --features mock --bin llama_chat_web` or `TEST_MODE=true cargo run --bin llama_chat_web`.

Tool calling: 120+ total tools organized via a tool catalog system to minimize prompt tokens. Only 24 core tools (~4K tokens) are sent in the system prompt. Desktop tools (90+) and MCP tools are discoverable on demand via `list_tools(category)` and `get_tool_details(tool_name)`. Core tools (always in prompt): read_file, write_file, edit_file, undo_edit, insert_text, search_files, find_files, lsp_query, execute_python, execute_command, list_directory, browser_search, browser_navigate, browser_get_text, browser_click, open_url, git_status, git_diff, git_commit, check_background_process, list_background_processes, send_telegram, spawn_agent, list_skills, use_skill, set_response_style. Tool input validation: 22 core tools validated against schemas before execution. Parallel execution: read-only tools run concurrently via std::thread::scope (max 10), write tools run serially. Tool output: map-reduce summarization for large outputs (>8K chars) using sub-agent with reusable CUDA context. Desktop automation tools: click_screen, type_text, press_key, move_mouse, scroll_screen, OCR, UI tree, window management — uses `enigo` crate. Screenshots auto-resized to max 1280px + JPEG compressed for vision.

Web browsing: The agent uses a native WebView (WebView2 on Windows, WKWebView on macOS) for ALL web search and browsing — no HTTP clients, no headless browsers. The WebView bypasses bot detection because it behaves like a real browser. Desktop (Tauri): the browser panel is a child WebView of the main Tauri window, overlaid inside the app UI. Web/server mode (no Tauri): the agent spawns a standalone wry+tao WebView window — a real native browser window that pops up separately. Iframes cannot be used in web mode because sites like Google block embedding via X-Frame-Options.

Cloud providers: `src/web/providers/` — openai_compat.rs (13 presets: Groq, Gemini, SambaNova, Cerebras, OpenRouter, Together, DeepSeek, Mistral, Fireworks, xAI, NVIDIA NIM, Hugging Face, Cloudflare + user-defined custom providers), claude_code.rs (CLI via `node cli.js`), codex.rs (CLI via `node codex.js`). Agentic tool loop in openai_compat.rs: sends tools as OpenAI function definitions → model returns tool_calls → `execute_openai_tool()` runs locally → result injected → loop (max 20 iterations). API keys stored in `provider_api_keys` JSON column in config DB, with env var fallback per provider. On Windows, Claude Code and Codex use `node <path>/cli.js` directly (not .cmd wrappers, which break with tokio::process::Command).

MCP (Model Context Protocol): External tool servers connected via Settings > MCP Servers. Uses JSON-RPC 2.0 over stdio (child process). Tools are namespaced as `mcp__<server>__<tool>` and discoverable via `list_tools("mcp")` / `get_tool_details()`. MCP servers auto-connect on worker startup. Backend: `src/web/mcp/` (McpManager with persistent tokio runtime). Uses `rmcp` crate.

Vision: Always compiled (all build scripts include `vision` feature). Models with mmproj GGUF files can see images — paste/drop in chat or use `take_screenshot` tool. Vision pipeline: `MtmdContext` + `MtmdBitmap` → `eval_chunks()`. Auto-detects mmproj GGUF in model directory.

Tool Systems — IMPORTANT DISTINCTION: There are TWO separate tool systems. (1) The app's built-in native tools (`src/web/native_tools/` + `src/web/desktop_tools/`) — called BY the loaded LLM agent during chat conversations, require a model to be loaded. (2) MCP server tools — external tools via the Model Context Protocol. Claude Code uses its own MCP servers (desktop-tools, chrome-devtools) which work WITHOUT any model loaded in our app because Claude IS the model. Our app's desktop tools are NOT the same as Claude's `computer` tool — they share similar functionality but are independent implementations.

OCR engines: Windows: Tesseract (auto-download, ~95% accuracy) → ocrs (Rust-native, ~70%) → WinRT (~40%). macOS: Vision framework → Tesseract → ocrs. Linux: Tesseract → ocrs. All engines get 2x image upscale. The agent can force a specific engine via the `engine` parameter: "tesseract", "ocrs", "native"/"winrt"/"vision", or "auto" (default). ocrs models are bundled in `assets/ocr-models/` (~12MB). Tesseract is auto-downloaded on first use via `tools/ensure-tesseract`.

Browser automation: use the Chrome DevTools MCP (chrome-devtools-mcp) for browser testing, NOT Playwright or the Claude Chrome extension. Install with: `claude mcp add chrome-devtools --scope user npx chrome-devtools-mcp@latest`. Use this to interact with the UI at http://localhost:14000.

Desktop app WebView debugging: The Tauri app exposes CDP on port 9222. Point chrome-devtools-mcp at port 9222 to automate/debug the actual desktop window. The Tauri WebView has `window.__TAURI_INTERNALS__` injected → `isTauriEnv()=true` → chat goes through `TauriChatTransport` (calls `invoke('generate_stream')`, receives Tauri events) instead of the WebSocket path.

Branch workflow: Always work on the `develop` branch. NEVER push to `master` directly — only merge develop → master after the user has tested and confirmed the changes work.

Common gotchas: use port 14000 for the UI (not 18080), keep backend running for Playwright tests, use Chrome DevTools MCP for browser automation (not Claude Chrome extension), must kill running llama_chat_web.exe before rebuilding on Windows (spawns 2 processes, kill both by PID), prefer existing modules rather than duplicating code. Adding untracked directories (vendor/, node_modules/) inside the project invalidates cargo fingerprints → forces full CUDA rebuild. Windows Git `link.exe` conflict: `C:\Program Files\Git\usr\bin\link.exe` shadows MSVC's linker — `scripts/ensure-vs.cmd` strips it from PATH automatically.

Shell execution (Windows): `command.rs` uses direct execution first (`Command::new`), PowerShell fallback for builtins. `needs_shell()` detects operators (|, >, &&, ;) → routes through PowerShell. CRITICAL: `stdin(Stdio::null())` on ALL child processes — without this, children inherit the worker's IPC stdin pipe and MSYS2 tools hang forever. CRITICAL: ALL spawned processes MUST use `CREATE_NO_WINDOW` flag (0x08000000) via `hide_cli_window()` or `.creation_flags(0x08000000)` — without this, a terminal window flashes on screen every time a process is spawned. Streaming command output: `execute_command_streaming()` uses `cmd /C "... 2>&1"` + byte-level pipe reading.

GPU layer safety: NEVER set ngl (n_gpu_layers) higher than the model's actual block_count. ngl > n_layers causes llama_decode to hang on hybrid MoE+DeltaNet models with large context. The vram_calculator reads GGUF block_count and caps automatically. Use ngl = n_layers (e.g., 40 for Qwen3.5's 40 layers), NOT 99.

Backend (src/web/): chat/ is the inference pipeline (generation.rs token loop, templates.rs prompt formatting, jinja_templates.rs Jinja2 rendering via minijinja, command_executor.rs tool execution, tool_tags.rs per-model tag config, stop_conditions.rs). routes/ has HTTP/WebSocket handlers. database/ is SQLite persistence. models.rs defines shared types (SamplerConfig, SharedLlamaState, ChatRequest/Response). config.rs loads assets/config.json. model_manager.rs handles model loading/unloading with VRAM-based GPU layer calculation. native_tools/mod.rs dispatches 24 core tools + tool catalog + 90+ desktop tools on demand. Command execution split: command.rs (shell exec, streaming, CWD/env persistence) + background.rs (background process tracking). Chat pipeline split: command_executor.rs + tool_parser.rs + loop_detection.rs + sub_agent.rs + tool_defs.rs. mcp/ handles MCP server connections. browser.rs is the headless Chrome singleton. websocket.rs handles WebSocket streaming. gguf_utils.rs extracts model metadata. vram_calculator.rs computes optimal GPU layer count.

Chat pipeline: WebSocket message → load config + model → resolve system prompt → format with chat template (ChatML/Mistral/Llama3/Gemma) using model-specific tool tags → tokenize → create context with KV cache on GPU if gpu_layers > 0 → generate tokens in loop → check stop conditions → detect and execute commands (regex on tool tags) → inject output back into context → stream tokens to frontend → log to SQLite.

Key types: SharedLlamaState (Arc<Mutex<Option<LlamaModelState>>>) wraps the loaded model. ConversationLogger writes to SQLite per-conversation. ToolTags (exec_open/close, output_open/close) are per-model (Qwen uses `<tool_call>`/`</tool_call>`, Mistral uses `[TOOL_CALLS]`/`[/TOOL_CALLS]`, GLM uses `<tool_call>`/`<|end_of_box|>`, Harmony uses `<|start|>tool<|message|>`/`<|end|>`, default uses SYSTEM.EXEC). SamplerConfig holds all inference params plus model_path and model_history.

Dynamic tool_tags pipeline: Tool tags are resolved dynamically from GGUF model metadata + presets without hardcoded model-specific branches. Backend: `tool_tags.rs` resolves tags via `get_tool_tags_for_model()` (exact match → fuzzy match → default), then `build_model_exec_regex()` builds detection regex from the resolved tags. Command detection in `command_executor.rs` uses FORMAT_PRIORITY (6 detectors): model_specific → exec → llama3 → harmony → mistral_bracket → mistral_json. Frontend: `toolFormatUtils.ts` resolves tags client-side, `toolSpanCollectors.ts` builds collectors dynamically from resolved tags. Adding new model tool tag support: update `tool_tags.rs` MODEL_TAG_MAP (backend) and `modelPresets.ts` MODEL_TOOL_TAGS (frontend) — see SYNC comments.

Common pitfall: Some models use DIFFERENT special tokens for open vs close tags. Example: GLM-4 opens with `<tool_call>` (token 151352) but closes with `<|end_of_box|>` (different token), NOT `</tool_call>` (token 151353). If the close tag is wrong, the tool call regex never matches and the model hallucinates fake tool responses.

Harmony model support (gpt-oss-20b): Detected via `<|start|>` + `<|end|>` + `<|channel|>` in the Jinja2 template. Uses channel-based turn structure: `<|start|>assistant<|channel|>analysis<|message|>` for reasoning, `to=tool_name code<|message|>{JSON}<|call|>` for tool calls, `<|start|>tool<|message|>...<|end|>` for tool output, and `<|channel|>final<|message|>` for the user-facing response. Frontend parser in useMessageParsing.ts (parseHarmonyContent) extracts ordered segments preserving chronological flow.

View modes and model output: RAW view shows `message.content` unmodified — never strip or "fix" model output there. MD and TXT views apply presentation cleanup (EOS tokens, vision markers, tool tags → widgets) via `useMessageParsing`. Do NOT add cleanup to hide model mistakes — if the model generates garbage, that's visible in RAW view intentionally.

Frontend (src/): Atomic design — atoms/ (Button, Dialog, etc.), molecules/ (MessageInput, ThinkingBlock, CommandExecBlock, ToolCallBlock), organisms/ (ModelSelector, SettingsModal, Sidebar), templates/ (ChatInputArea, MessagesArea). Key hooks: useChat (messaging orchestration), useModel (model lifecycle), useMessageParsing (extracts thinking/commands/tool calls, handles Harmony format), useToolExecution (tool call parsing with MAX_TOOL_ITERATIONS=20), useSettings (sampler config). Utils: chatTransport (HTTP/WS abstraction), toolParser (multi-format tool call extraction).

Model auto-configuration: When loading a GGUF model, gguf_utils.rs extracts general.sampling.* keys for optimal parameters. Fallback presets in src/config/modelPresets.ts keyed by general.name. Priority: GGUF embedded params → preset lookup → defaults. All model file paths, recommended configs, tool formats, and test results are documented in docs/MODEL_CONFIGURATIONS.md. Model GGUF files live under E:/.lmstudio/ (subdirs: lmstudio-community/, Mungert/, etc.). When the user asks to configure, research, or set up a new model: (1) read the GGUF general.name with `python -c "from gguf import GGUFReader; ..."` or `npm run inspect-gguf <path>`, (2) look up recommended params from the HuggingFace model card, (3) add a preset entry to src/config/modelPresets.ts keyed by exact general.name, (4) append an entry to docs/MODEL_CONFIGURATIONS.md.

Out-of-process worker: The model runs in a child process (`llama_chat_web.exe --worker`) spawned by the web server. Communication is JSON Lines over stdin/stdout pipes. Key files: `src/web/worker/` — `ipc_types.rs` (protocol types), `worker_main.rs` (child entry point, 3-thread design), `worker_bridge.rs` (server-side `WorkerBridge` abstraction), `process_manager.rs` (spawn/kill/restart). `SharedWorkerBridge = Arc<WorkerBridge>` is passed to all route handlers. Force-unload (`POST /api/model/hard-unload`) kills the child process so the OS reclaims ALL VRAM/RAM, then auto-restarts a fresh worker.

Tools (tools/ directory): Standalone Rust utilities, each with their own Cargo.toml. "tools/ensure-cmake" auto-downloads CMake if missing. "tools/start-dev" kills old processes and launches backend + Vite. GPU backend via `--gpu cuda|vulkan|cpu`. Quick commands: `npm run start:cuda`, `npm run start:vulkan`, `npm run start:cpu`, or `npm start`. "tools/print-metadata" inspects GGUF model files (`npm run inspect-gguf <path-to-gguf>`).

llama-cpp-rs submodule: Embedded as git submodule at `deps/llama-cpp-rs` (nested: `deps/llama-cpp-rs/llama-cpp-sys-2/llama.cpp`). Cargo.toml uses path deps: `path = "deps/llama-cpp-rs/llama-cpp-2"` and `llama-cpp-sys-2`. To update llama.cpp: `cd deps/llama-cpp-rs/llama-cpp-sys-2/llama.cpp && git pull`. To pull upstream Rust binding fixes: `cd deps/llama-cpp-rs && git pull`. Vision/mtmd disabled by default — gated behind `#[cfg(feature = "vision")]`. ALWAYS use `npm run cargo -- build ...` for manual cargo builds (not bare `cargo build`) — ensures cmake is available. Full CUDA rebuild ~27 min; incremental Rust-only ~20s.

Linting: `cargo clippy --bin llama_chat_web --features cuda` for Rust. `npm run lint` for frontend ESLint. Both should report zero warnings on clean code.

Performance: Token pipeline optimized to ~80-95 tok/s decode-only on RTX 4090. App reports internal `ctx.timings()` (decode-only, matches llama-server methodology) — NOT wall-clock. Key optimizations: KV cache reuse between turns (InferenceCache stores LlamaContext<'static> via transmute), system prompt caching, token prefix comparison to skip re-evaluating cached tokens, cancellation via CancellationFlag (Arc<AtomicBool>) checked every 4 tokens, per-token command check gated on close chars to skip ~90% of tokens. CRITICAL: context_size MUST fit in VRAM — CUDA VMM silently pages to RAM if oversubscribed → 70ms/tok instead of 5ms.

Sampler system: 11 sampler types in `create_sampler()` in `sampler.rs`. Chain ordering: penalties → DRY → top_n_sigma → filtering samplers → GBNF grammar → terminal (dist/greedy). DRY sampler requires &LlamaModel reference. LlamaSampler::temp() alone crashes — must chain with terminal sampler. GBNF lazy grammar (`tool_grammar.rs`) constrains JSON tool calls — only activates when model outputs `{"name"` trigger.

Conversation compaction: Auto-compacts when conversation approaches 70% of available context. Map-reduce summarization with single reusable context (avoids CUDA fragmentation). Uses real tokenizer for token counting. Marks old messages as `compacted=1` in DB, inserts LLM-generated summary. KV cache cleared in-place after compaction. Auto-continue after compaction with original task context. Key: `compaction.rs`, `conversation.rs:compact_messages()`.

Context-saving features: Duplicate file read detection (returns stub on unchanged re-reads). Max 2000 lines default cap on read_file. LRU file content cache (50 entries, 25MB, mtime-validated). Content-based binary detection. Token analysis API: GET /api/conversations/{id}/token-analysis.

Streaming command timeout: Commands have a 120s inactivity timeout (resets on output) AND a 120s wall-clock timeout (absolute max). Shell state persistence: CWD persists across compound commands. Environment variables from `export VAR=value` / `set VAR=value` captured and injected into subsequent commands.

Skills system: Markdown-based prompt templates in `skills/` directory. `list_skills` / `use_skill` tools with {{variable}} template substitution.

Security: Destructive command warnings (rm -rf, git reset --hard, DROP TABLE). Command injection detection (blocks curl|base64, eval+wget). Tool schema validation for 22 core tools. Desktop abort mechanism (POST /api/desktop/abort). Windows sleep prevention during generation (SetThreadExecutionState).

conversation_context table: Caches system_prompt_text/tokens and tool_definitions_json/tokens per conversation. Populated lazily on first generation via content hash dirty-checking. Key finding: tool definitions consume ~14K tokens (~43% of 32K context).

messages table row structure: One row per generation turn or tool execution. role=user → user input. role=assistant → model's generated text + tool call declaration, stored as JSON: {"content": "text...", "tool_calls": [...]}. role=tool → tool execution result, stored as plain text "tool_name\n\nresult". The model cannot write text after a tool call in the same turn — it stops at the tool call, the tool runs (new role=tool row), then a new generation turn begins (new role=assistant row).

Desktop app (Tauri): System tray (minimize to tray on close), app menu (File/Edit with Ctrl+N/Q/,), window state persistence, CSP security, auto-updater plugin, deep links (llamachat://), `CREATE_NO_WINDOW` on all subprocess spawns, proper app data directory (`%APPDATA%/com.llamachat.desktop/` on Windows, `~/Library/Application Support/com.llamachat.desktop/` on macOS). Two separate databases: the web server uses `assets/llama_chat.db`; the Tauri desktop app uses the app data directory DB. Cloud provider streaming in Tauri mode goes through the `stream_provider` Tauri command (src/main.rs) NOT the HTTP `/api/providers/*/stream` route — `useCloudChat.ts` detects `isTauriEnv()` and uses `invoke('stream_provider')` + Tauri events instead of SSE fetch. If you add cloud provider features, update both the HTTP route handler (crates/llama-chat-web/src/routes/providers.rs) AND the Tauri command (src/main.rs `stream_provider`).

macOS-specific: Build with `npm run tauri:build:metal` (requires `brew install sccache`). The git submodule at `deps/llama-cpp-rs` points to a Windows path — on Mac, override with `git config submodule.deps/llama-cpp-rs.url https://github.com/utilityai/llama-cpp-rs.git` then clone manually. Native file picker crashes in web server mode — `pick-file` uses `osascript` fallback on macOS. System info (`/api/system/usage`) uses `sysctl` for RAM/cores on macOS (unified memory reported as both RAM and VRAM). Desktop-tools MCP server: `src/mcp_server.rs` builds as standalone binary `mcp_desktop_tools` — register with `claude mcp add desktop-tools -- ./target/release/mcp_desktop_tools`.

Cancel generation: Sets cancel flag → `execute_command_streaming()` kills subprocess within 200ms. Worker waits up to 3s for cancelled thread to finish before accepting new Generate command.

WebSocket /ws/status: Persistent health-check WebSocket with 20s ping keep-alive. Frontend ConnectionProvider (ConnectionContext.tsx) reconnects with exponential backoff (500ms → 5s). Red ConnectionBanner appears when server disconnects. Skipped in Tauri mode.

System monitor: SystemResourcesContext polls at 10s background rate, switches to 3s when sidebar is open (setMonitorActive). Graphs render instantly on sidebar open because data is pre-populated.

Model testing results (RTX 4090 24GB, 6-test agent suite): PASS 6/6: Devstral-Small-2-2512, Devstral-Small-2507, Magistral-Small-2509, Qwen3-Coder-30B-A3B-1M, Ministral-3-14B-Reasoning. NEAR-PASS 5.5/6: GLM-4.7-Flash. Config is critical — wrong temp/context can drop a 6/6 model to 1/6. Models <14B generally fail. See docs/MODEL_CONFIGURATIONS.md.

REST API (headless/programmatic usage): GET /api/model/status — check if a model is loaded. POST /api/model/load — load a GGUF model: {"model_path": "E:/path/to/model.gguf", "gpu_layers": 99, "context_size": 32768, "flash_attention": true, "cache_type_k": "q8_0", "cache_type_v": "q8_0"}. For large MoE models requiring 110K+ context, use TurboQuant KV cache: "gpu_layers": 40, "context_size": 114688, "cache_type_k": "turbo2", "cache_type_v": "turbo3" (note: gpu_layers must not exceed the model's actual block_count). POST /api/model/unload — unload current model. POST /api/model/hard-unload — kill worker process (reclaim all VRAM). POST /api/chat/stream — SSE streaming chat: {"message": "your prompt", "conversation_id": null}. Returns data: {"token": "...", "tokens_used": N} events, ends with data: [DONE]. Omit conversation_id or set null to create a new conversation. GET /api/conversations — list all conversations. GET /api/config / POST /api/config — read/update sampler configuration. GET /api/browse?path=... — browse model files on disk. GET /health — health check. Note: browser tools fall back to curl in web mode (no Tauri WebView) — works for static sites, fails for Google/JS-heavy sites.

Watchdog and tool execution: The token loop has a 10s watchdog thread that detects CUDA deadlocks (sample()/decode() stalls) and kills the worker via std::process::exit(42). The watchdog is PAUSED during tool execution (`watchdog_paused` AtomicBool) because tools like browser_search can take 20+ seconds. After tool execution, the heartbeat is reset.

Tool injection pipeline: When a tool call is detected during generation, the output is injected directly into the KV cache mid-generation (no context restart). Injection path: `check_and_execute_command_with_tags()` → execute tool → tokenize result with `<tool_response>` tags → `inject_output_tokens()` → `sampler.accept_many()` → 50ms CUDA settle pause → continue generating. Vision tool results use a separate path via `inject_tool_response_with_vision()` with MtmdContext.

Worker memory eviction: On memory-constrained machines, two co-resident model copies OOM the GPU — surfaces as "Decode Error -3" mid-decode. `crates/llama-chat-web/src/worker_pool.rs` auto-evicts idle workers before loading a new model when capacity is tight. Hooked into `spawn_worker_with_options` and `handle_post_model_load`.

Headless architecture (TODO): The backend should control the entire agentic flow without frontend involvement. Currently the frontend (`useGenerationStream.ts`) handles auto-continue for finish reasons: `length`, `yn_continue`, `loop_recovery`, `tool_continue`, `cuda_deadlock`, `infinite_loop`. Goal: move all auto-continue logic into the backend so the app works fully headless via the REST API.

Deprecated/removed: `web_search_provider` and `web_search_api_key` config fields — columns remain in DB schema for migration compatibility but removed from all Rust structs. `web_search.rs` (DuckDuckGo HTTP search) — deleted. The only web search mechanism is `browser_search` via the Tauri WebView. The `web_search` tool name is aliased to `browser_search` in the OpenAI-compat provider for backward compatibility.

TTS Notifications: Claude Code hooks in `.claude/hooks/` trigger TTS on events. Unified dispatcher `hook.py` handles session/submit/notification/stop events. Current engine: "homer" (Qwen3-TTS 1.7B voice clone, Spanish LATAM). Venv at `E:/repo/faster-qwen3-tts/.venv/`. The stop hook reads `last_assistant_message`, strips markdown, and speaks it aloud.

CodeGraph: This project has a CodeGraph MCP server configured. Use `codegraph_search` to find where a symbol is defined. Use `codegraph_callers`/`codegraph_callees` to find what calls a function or what it calls. Use `codegraph_impact` to see what would break if you change a symbol. Use `codegraph_context` for a focused view of a task area. Prefer codegraph for structural questions; use grep/read for literal text or comments. Trust codegraph results — they come from a full AST parse. If `.codegraph/` doesn't exist, the index hasn't been built — ask the user if they want to run `codegraph init -i`.
