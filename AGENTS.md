AGENTS GUIDE

This is the short canonical reference for agents (Claude Code, OpenAI Agents, etc.) working in this repo.

What this is: Multi-provider AI chat app (React + Rust). Supports local models (llama.cpp GGUF), CLI-backed providers (Claude Code, Codex), and 13+ OpenAI-compatible cloud providers (Groq, Gemini, SambaNova, Cerebras, Mistral, etc.) with agentic tool loop. Core local flow: load GGUF model → auto-calculate GPU layers from available VRAM → extract Jinja2 chat template + model metadata from GGUF → resolve model-specific tool format and sampler presets → format prompts via Jinja template → generate tokens with configurable sampler chain (11 types) → stream over WebSocket → detect tool call patterns in output (`<tool_call>`, `[TOOL_CALLS]`, `SYSTEM.EXEC`, etc.) → execute native tools (file I/O, shell, web search/fetch, Python) → inject results back into context for multi-turn agentic loops. Cloud providers: OpenAI-compatible `/v1/chat/completions` API with function calling → local tool execution → result injection → loop (max 20 iterations). KV cache reused between conversation turns for speed. Conversations and config persisted in SQLite. Model runs in an out-of-process worker (child process with JSON Lines IPC) — kill worker to reclaim all VRAM instantly. Runs as web app (Vite + Rust backend) or Tauri desktop app.

Default development (prefer Tauri commands — they handle both frontend and backend together):
- Desktop app (preferred): "npm run dev:auto:desktop" (automatic GPU detection) or "npm run tauri:dev" (CPU-only)
- Web app (fallback): "npm run dev:auto" (automatic GPU detection) or "npm run dev:cuda" (manual CUDA)
- CRITICAL: "npm run dev:cpu" is CPU-ONLY — it does NOT pass --features cuda. Model will run at ~5 tok/s instead of ~1000 tok/s. ALWAYS use "npm run dev:cuda" or "npm run dev:auto" for GPU acceleration.

Web app runs Vite on port 14000 with Rust backend on port 18080. Access via http://localhost:14000. Desktop app opens native window. When building/running, prefer Tauri commands over running backend and Vite separately.

GPU acceleration: "npm run dev:auto" (web) or "npm run dev:auto:desktop" automatically detect the best setup (Metal on macOS, CUDA on Windows, CPU fallback). Manual options: "npm run dev:metal"/"npm run tauri:dev:metal", "npm run dev:cuda"/"npm run tauri:dev:cuda".

Frontend alternatives: "npm run build" for production build. If the backend is already running, start Vite only with "npx vite --host --port 4000".

Backend alternatives: "cargo run --bin llama_chat_web" if Vite is already running. Rust builds: "cargo build --lib" and "cargo build --bin llama_chat_web".

Desktop app: "cargo tauri dev" for hot reload, "cargo tauri build" for production desktop. IMPORTANT: Always use "npm run tauri:build" (NOT "tauri:build:cuda") for the installer. CUDA DLLs must NOT be bundled — the user installs the CUDA toolkit themselves. The app detects NVIDIA GPU + missing CUDA at runtime and shows an install link in the load model modal. Bundling CUDA bloats the installer from ~20MB to ~654MB.

CMake: required by llama-cpp-sys-2. All npm scripts route through tools/ensure-cmake, a standalone Rust tool that auto-downloads a portable CMake if it's not on PATH. This solves the chicken-and-egg problem where llama-cpp-sys-2's build script needs cmake before our build.rs runs. The ensure-cmake tool checks: (1) "cmake" on PATH, (2) well-known install locations (C:\Program Files\CMake, /usr/local/bin, /opt/homebrew/bin), (3) cached download in target/cmake/, (4) fresh download from GitHub releases. When cmake is found at an absolute path not on PATH, it injects the directory into the child process's PATH.

Docker:
- CPU: "docker build -f Dockerfile.test-cmake -t llama-cpu ." then "docker run -p 8000:8000 -v /path/to/models:/app/models -v ./assets:/app/assets llama-cpu"
- CUDA: "docker build -f Dockerfile.cuda -t llama-cuda ." then "docker run --gpus all -p 8000:8000 -v /path/to/models:/app/models -v ./assets:/app/assets llama-cuda"
- Mount models to /app/models (the browse endpoint only allows /app/models and /app paths)
- CUDA requires NVIDIA Container Toolkit on the host

Testing: "npm test" (Playwright E2E; backend must be running on 18080). UI/headed/debug variants: "npm run test:ui", "npm run test:headed", "npm run test:debug", "npm run test:docker". Unit tests: "cargo test". Single browser: "npx playwright test --project=chromium".

Mock mode for tests: build or run with the "mock" feature. Example: "cargo build --features mock --bin llama_chat_web" or "TEST_MODE=true cargo run --bin llama_chat_web".

Tool calling: 120+ total tools organized via a **tool catalog system** to minimize prompt tokens. Only 24 core tools (~4K tokens) are sent in the system prompt. Desktop tools (90+) and MCP tools are discoverable on demand via `list_tools(category)` and `get_tool_details(tool_name)`. Core tools (always in prompt): read_file, write_file, edit_file, undo_edit, insert_text, search_files, find_files, lsp_query, execute_python, execute_command, list_directory, web_search, web_fetch, open_url, git_status, git_diff, git_commit, check_background_process, list_background_processes, send_telegram, spawn_agent, list_skills, use_skill, set_response_style. Tool input validation: 22 core tools validated against schemas before execution (required params, type checking with lenient coercion). Parallel execution: read-only tools (read_file, search_files, web_search, etc.) run concurrently via std::thread::scope (max 10), write tools run serially. Tool output: map-reduce summarization for large outputs (>8K chars) using sub-agent with reusable CUDA context, one-liner summaries prepended to truncated output. Desktop automation tools (src/web/desktop_tools/): click_screen, type_text, press_key, move_mouse, scroll_screen, OCR, UI tree, window management — uses `enigo` crate. Screenshots auto-resized to max 1280px + JPEG compressed for vision. Desktop abort: POST /api/desktop/abort + red Abort button in UI. The Chrome browser is a lazy singleton with 5-minute idle timeout.
Brave web_search: users can store their Brave API key in the app settings (persisted in SQLite config as plaintext). The backend will also fall back to `BRAVE_SEARCH_API_KEY` if set.

Cloud providers: `src/web/providers/` — openai_compat.rs (13 presets: Groq, Gemini, SambaNova, Cerebras, OpenRouter, Together, DeepSeek, Mistral, Fireworks, xAI, NVIDIA NIM, Hugging Face, Cloudflare + user-defined custom providers), claude_code.rs (CLI via `node cli.js`), codex.rs (CLI via `node codex.js`). Agentic tool loop in openai_compat.rs: sends tools as OpenAI function definitions → model returns tool_calls → `execute_openai_tool()` runs locally → result injected → loop (max 20 iterations). API keys stored in `provider_api_keys` JSON column in config DB, with env var fallback per provider. Provider list/stream endpoints in `routes/providers.rs`. On Windows, Claude Code and Codex use `node <path>/cli.js` directly (not .cmd wrappers, which break with tokio::process::Command).

MCP (Model Context Protocol): External tool servers can be connected via Settings > MCP Servers. Uses JSON-RPC 2.0 over stdio (child process). Tools are namespaced as `mcp__<server>__<tool>` and discoverable via `list_tools("mcp")` / `get_tool_details()`. MCP servers auto-connect on worker startup. Backend: `src/web/mcp/` (McpManager with persistent tokio runtime). Frontend: MCP server config UI in AppSettingsModal. Uses `rmcp` crate.

Vision: Always compiled (all build scripts include `vision` feature). Models with mmproj GGUF files can see images — paste/drop in chat or use `take_screenshot` tool. Vision pipeline: `MtmdContext` + `MtmdBitmap` → `eval_chunks()`. Auto-detects mmproj GGUF in model directory.

Tool Systems — IMPORTANT DISTINCTION: There are TWO separate tool systems. (1) The app's built-in native tools (`src/web/native_tools/` + `src/web/desktop_tools/`) — these are called BY the loaded LLM agent during chat conversations. They require a model to be loaded because the model decides when/how to call them. The agent discovers desktop tools via `list_tools("desktop")`. Core tools (read_file, write_file, execute_command, web_search, take_screenshot, ocr_screen, etc.) are always in the system prompt. (2) MCP server tools — these are external tools connected via the Model Context Protocol. Claude Code (the CLI) uses its own MCP servers (desktop-tools, chrome-devtools) which work WITHOUT any model loaded in our app because Claude IS the model. Our app can also connect to external MCP servers via Settings > MCP. Don't confuse the three: (a) Our app's built-in desktop tools — called by the local LLM agent, needs model loaded. (b) Claude Code's MCP desktop-tools server — called by Claude directly, no model needed. (c) Claude Code's own built-in `computer` tool — this is Claude's native computer control, completely separate from both our app and the desktop MCP. Our app's desktop tools are NOT the same as Claude's `computer` tool. They share similar functionality (screenshot, click, type) but are independent implementations in different codebases.

OCR engines: The app includes multiple OCR engines with auto-fallback priority. Windows: Tesseract (auto-download via ensure-tesseract, ~95% accuracy) → ocrs (Rust-native, bundled, ~70%) → WinRT (built-in, ~40%). macOS: Vision framework (built-in, excellent) → Tesseract (if installed) → ocrs. Linux: Tesseract (AppImage auto-download) → ocrs. All engines get 2x image upscale for accuracy. The agent can force a specific engine via the `engine` parameter: "tesseract", "ocrs", "native"/"winrt"/"vision", or "auto" (default). ocrs models are bundled in `assets/ocr-models/` (~12MB). Tesseract is auto-downloaded on first use via `tools/ensure-tesseract`.

Browser automation: use the Chrome DevTools MCP (chrome-devtools-mcp) for browser testing, NOT Playwright or the Claude Chrome extension. Install with: "claude mcp add chrome-devtools --scope user npx chrome-devtools-mcp@latest". Use this to interact with the UI at http://localhost:14000 for testing models, chat, and features.

Common gotchas: use port 14000 for the UI (not 18080), keep backend running for Playwright tests, use Chrome DevTools MCP for browser automation (not Claude Chrome extension), must kill running llama_chat_web.exe before rebuilding on Windows (Access denied — now spawns 2 processes, kill both), prefer existing modules rather than duplicating code when editing web routes or chat logic. Adding untracked directories (vendor/, node_modules/) inside the project invalidates cargo fingerprints → forces full CUDA rebuild.

Shell execution (Windows): `command.rs` uses direct execution first (`Command::new`), PowerShell fallback for builtins. `needs_shell()` detects operators (|, >, &&, ;) → routes through PowerShell. `$` escaped to backtick-$ for PowerShell. CRITICAL: `stdin(Stdio::null())` on ALL child processes — without this, children inherit the worker's IPC stdin pipe and MSYS2 tools (wc, grep, etc.) hang forever. Streaming command output: `execute_command_streaming()` uses `cmd /C "... 2>&1"` + byte-level pipe reading for live output (progress bars, cargo/npm output). Callback sends each line via token_sender → IPC → WebSocket.

Architecture:

GPU layer safety: NEVER set ngl (n_gpu_layers) higher than the model's actual block_count. ngl > n_layers offloads the output/embedding layer to GPU, which causes llama_decode to hang on Qwen3.5 (hybrid MoE+Mamba2/DeltaNet) with large context (262K). The vram_calculator reads GGUF block_count and caps automatically. model_manager.rs also caps explicitly-requested gpu_layers. Use ngl = n_layers (e.g., 40 for Qwen3.5's 40 layers), NOT 99.

Backend (src/web/): chat/ is the inference pipeline (generation.rs token loop with conditional KV cache GPU offload, templates.rs prompt formatting, jinja_templates.rs Jinja2 rendering via minijinja, command_executor.rs tool execution, tool_tags.rs per-model tag config, stop_conditions.rs). routes/ has HTTP/WebSocket handlers (chat, config, model, conversations, tools, files, health, logs, system, frontend_logs, static_files). database/ is SQLite persistence (conversations, messages, config, migration). models.rs defines shared types (SamplerConfig, SharedLlamaState, ChatRequest/Response). config.rs loads assets/config.json and resolves system prompts. model_manager.rs handles model loading/unloading with VRAM-based GPU layer calculation. native_tools/mod.rs dispatches 24 core tools + tool catalog (list_tools/get_tool_details) + 90+ desktop tools on demand. Sub-modules: mcp_tools.rs, telegram.rs, web_fetch.rs. Command execution split: command.rs (shell exec, streaming, CWD/env persistence) + background.rs (background process tracking). Chat pipeline split: command_executor.rs + tool_parser.rs + loop_detection.rs + sub_agent.rs + tool_defs.rs (compact definitions). desktop_tools.rs has mouse/keyboard/scroll simulation via enigo. mcp/ handles MCP server connections and tool proxying. browser.rs is the headless Chrome singleton. websocket.rs handles WebSocket streaming. gguf_utils.rs extracts model metadata for auto-configuration. vram_calculator.rs computes optimal GPU layer count.

Chat pipeline: WebSocket message -> load config + model -> resolve system prompt (agentic/custom/model-default) -> format with chat template (ChatML/Mistral/Llama3/Gemma) using model-specific tool tags -> tokenize -> create context with KV cache on GPU if gpu_layers > 0 -> generate tokens in loop -> check stop conditions -> detect and execute commands (regex on tool tags) -> inject output back into context -> stream tokens to frontend -> log to SQLite.

Key types: SharedLlamaState (Arc<Mutex<Option<LlamaModelState>>>) wraps the loaded model. ConversationLogger writes to SQLite per-conversation. ToolTags (exec_open/close, output_open/close) are per-model (Qwen uses <tool_call>/<\/tool_call>, Mistral uses [TOOL_CALLS]/[/TOOL_CALLS], GLM uses <tool_call>/<|end_of_box|>, Harmony uses <|start|>tool<|message|>/<|end|>, default uses SYSTEM.EXEC). SamplerConfig holds all inference params plus model_path and model_history.

Dynamic tool_tags pipeline: Tool tags are now resolved dynamically from GGUF model metadata + presets and flow through the entire stack without hardcoded model-specific branches. Backend: `tool_tags.rs` resolves tags via `get_tool_tags_for_model()` (exact match → fuzzy match → default), then `build_model_exec_regex()` builds detection regex from the resolved tags. Command detection in `command_executor.rs` uses FORMAT_PRIORITY (6 detectors): model_specific → exec → llama3 → harmony → mistral_bracket → mistral_json. Frontend: `toolFormatUtils.ts` resolves tags client-side, `toolSpanCollectors.ts` builds collectors dynamically from resolved tags. Adding new model tool tag support: update `tool_tags.rs` MODEL_TAG_MAP (backend) and `modelPresets.ts` MODEL_TOOL_TAGS (frontend) — see SYNC comments.

Common pitfall: Some models use DIFFERENT special tokens for open vs close tags. Example: GLM-4 opens with `<tool_call>` (token 151352) but closes with `<|end_of_box|>` (different token), NOT `</tool_call>` (token 151353). If the close tag is wrong, the tool call regex never matches and the model hallucinates fake tool responses.

Harmony model support (gpt-oss-20b): Detected via `<|start|>` + `<|end|>` + `<|channel|>` in the Jinja2 template. Uses channel-based turn structure: `<|start|>assistant<|channel|>analysis<|message|>` for reasoning, `to=tool_name code<|message|>{JSON}<|call|>` for tool calls, `<|start|>tool<|message|>...<|end|>` for tool output, and `<|channel|>final<|message|>` for the user-facing response. Frontend parser in useMessageParsing.ts (parseHarmonyContent) extracts ordered segments preserving chronological flow.

View modes and model output: RAW view shows `message.content` unmodified — never strip or "fix" model output there. MD and TXT views apply presentation cleanup (EOS tokens, vision markers, tool tags → widgets) via `useMessageParsing`. Do NOT add cleanup to hide model mistakes — if the model generates garbage (hallucinated chat template tags, broken tool calls, etc.), that's the model's fault and should be visible. The RAW view exists for exactly this purpose.

Frontend (src/): Atomic design — atoms/ (Button, Dialog, etc.), molecules/ (MessageInput, ThinkingBlock, CommandExecBlock, ToolCallBlock), organisms/ (ModelSelector, SettingsModal, Sidebar), templates/ (ChatInputArea, MessagesArea). Message widgets: ThinkingBlock (collapsible reasoning with streaming support — detects unclosed `<think>` tags during streaming), CommandExecBlock (tool execution with animated "Executing Tool..." state when output is null, collapsed details with CLI-style command summary when complete), ToolCallBlock (JSON tool calls). Key hooks: useChat (messaging orchestration), useModel (model lifecycle), useMessageParsing (extracts thinking/commands/tool calls from message content, handles Harmony format), useToolExecution (tool call parsing with MAX_TOOL_ITERATIONS=20), useSettings (sampler config). Utils: chatTransport (HTTP/WS abstraction), toolParser (multi-format tool call extraction).

Model auto-configuration: When loading a GGUF model, gguf_utils.rs extracts general.sampling.* keys for optimal parameters. Fallback presets in src/config/modelPresets.ts keyed by general.name. Priority: GGUF embedded params -> preset lookup -> defaults. All model file paths, recommended configs, tool formats, and test results are documented in `docs/MODEL_CONFIGURATIONS.md`. Model GGUF files live under `E:/.lmstudio/` (subdirs: lmstudio-community/, Mungert/, etc.).

Out-of-process worker: The model runs in a child process (`llama_chat_web.exe --worker`) spawned by the web server. Communication is JSON Lines over stdin/stdout pipes. Key files: `src/web/worker/` — `ipc_types.rs` (protocol types), `worker_main.rs` (child entry point, 3-thread design), `worker_bridge.rs` (server-side `WorkerBridge` abstraction replacing `SharedLlamaState + GenerationQueue`), `process_manager.rs` (spawn/kill/restart). `SharedWorkerBridge = Arc<WorkerBridge>` is passed to all route handlers. Force-unload (`POST /api/model/hard-unload`) kills the child process so the OS reclaims ALL VRAM/RAM, then auto-restarts a fresh worker. The worker's `run_generation` uses a single-threaded tokio runtime — token forwarding uses a real OS thread (not tokio::spawn) because the generation loop has no yield points.

Tools (tools/ directory): Standalone Rust utilities, each with their own Cargo.toml. "tools/ensure-cmake" auto-downloads CMake if missing (also a library used by start-dev). "tools/start-dev" kills old processes and launches backend + Vite. GPU backend via `--gpu cuda|vulkan|cpu` (default cpu). Quick commands: `npm run start:cuda`, `npm run start:vulkan`, `npm run start:cpu`, or `npm start` (run existing binary). "tools/print-metadata" inspects GGUF model files (`npm run inspect-gguf <path-to-gguf>`).

llama-cpp-rs submodule: Embedded as git submodule at `deps/llama-cpp-rs` (nested: `deps/llama-cpp-rs/llama-cpp-sys-2/llama.cpp`). Origin: `https://github.com/utilityai/llama-cpp-rs.git`. Cargo.toml uses path deps: `path = "deps/llama-cpp-rs/llama-cpp-2"` and `llama-cpp-sys-2`. To update llama.cpp: `cd deps/llama-cpp-rs/llama-cpp-sys-2/llama.cpp && git pull`. To pull upstream Rust binding fixes: `cd deps/llama-cpp-rs && git pull`. Vision/mtmd disabled by default — gated behind `#[cfg(feature = "vision")]`. ALWAYS use `npm run cargo -- build ...` for manual cargo builds (not bare `cargo build`) — ensures cmake is available. Full CUDA rebuild ~27 min; incremental Rust-only ~20s. NEVER test with debug build — prompt eval 5+ min in debug vs <2s in release.

Linting: "cargo clippy --bin llama_chat_web --features cuda" for Rust. "npm run lint" for frontend ESLint. Both should report zero warnings on clean code.

Performance: Token pipeline optimized to ~80-95 tok/s decode-only on RTX 4090 (Qwen3.5-35B-A3B IQ4_XS, 32K ctx). App reports internal `ctx.timings()` (decode-only, matches llama-server methodology) — NOT wall-clock. Key optimizations: KV cache reuse between turns (InferenceCache stores LlamaContext<'static> via transmute), system prompt caching, token prefix comparison to skip re-evaluating cached tokens, cancellation via CancellationFlag (Arc<AtomicBool>) checked every 4 tokens, per-token command check gated on close chars (`>`, `]`, `}`) to skip ~90% of tokens. `build_context_params()` sets `no_perf(false)` to enable internal timing; `reset_timings()` called before each turn. Zero FFI overhead confirmed (Rust bindings = C decode speed). CRITICAL: context_size MUST fit in VRAM — CUDA VMM silently pages to RAM if oversubscribed → 70ms/tok instead of 5ms.

Sampler system: 11 sampler types in `create_sampler()` in `sampler.rs`. Chain ordering: penalties → DRY → top_n_sigma → filtering samplers → GBNF grammar → terminal (dist/greedy). DRY sampler requires &LlamaModel reference. LlamaSampler::temp() alone crashes — must chain with terminal sampler. GBNF lazy grammar (`tool_grammar.rs`) constrains JSON tool calls — only activates when model outputs `{"name"` trigger, transparent to XML format.

Conversation compaction: Auto-compacts when conversation approaches 70% of available context (context_size - overhead). Map-reduce summarization with single reusable context (avoids CUDA fragmentation). Uses real tokenizer for token counting (not chars/4 heuristic). Recompaction: if first pass insufficient, automatically recurses (max depth 2). Marks old messages as `compacted=1` in DB, inserts LLM-generated summary. KV cache cleared in-place after compaction (no context drop). Auto-continue after compaction with original task context. Frontend reloads messages from DB after compaction completes. Mid-task compaction available for long tool sessions. Key: `compaction.rs`, `conversation.rs:compact_messages()`.

Context-saving features: Duplicate file read detection (returns stub on unchanged re-reads). Max 2000 lines default cap on read_file. LRU file content cache (50 entries, 25MB, mtime-validated). Content-based binary detection (byte analysis, not just extension). Token analysis API: GET /api/conversations/{id}/token-analysis.

Streaming command timeout: Commands have a 120s inactivity timeout (resets on output) AND a 120s wall-clock timeout (absolute max, kills even chatty commands). Safety net after pipe disconnect prevents child.wait() blocking. Poll-based reap with 5s deadline. The model gets a message with the PID so it can adapt.

Shell state persistence: CWD persists across compound commands (`cd X && ...` updates process CWD). Environment variables from `export VAR=value` / `set VAR=value` captured and injected into subsequent commands.

Skills system: Markdown-based prompt templates in `skills/` directory. `list_skills` / `use_skill` tools with {{variable}} template substitution. Example skills: create-project, fix-errors.

Security: Destructive command warnings (rm -rf, git reset --hard, DROP TABLE). Command injection detection (blocks curl|base64, eval+wget). Edit file curly quote normalization. Tool schema validation for 22 core tools. Desktop abort mechanism (POST /api/desktop/abort). Windows sleep prevention during generation (SetThreadExecutionState).

conversation_context table: Caches system_prompt_text/tokens and tool_definitions_json/tokens per conversation. Populated lazily on first generation via content hash dirty-checking. Compaction uses real overhead from this table instead of hardcoded estimate. Key finding: tool definitions consume ~14K tokens (~43% of 32K context).

Desktop app (Tauri): System tray (minimize to tray on close), app menu (File/Edit with Ctrl+N/Q/,), window state persistence (excludes fullscreen), CSP security, auto-updater plugin, deep links (llamachat://), `CREATE_NO_WINDOW` on all subprocess spawns (`silent_command()` in `utils.rs`), proper app data directory (`%APPDATA%/com.llamachat.desktop/` on Windows, `~/Library/Application Support/com.llamachat.desktop/` on macOS), model loading progress bar via `with_progress_callback` (Windows only — uses private FFI field not available in upstream llama-cpp-rs). CRITICAL — two separate databases: the web server uses `assets/llama_chat.db`; the Tauri desktop app uses the app data directory DB. Cloud provider streaming in Tauri mode goes through the `stream_provider` Tauri command (src/main.rs) NOT the HTTP `/api/providers/*/stream` route — `useCloudChat.ts` detects `isTauriEnv()` and uses `invoke('stream_provider')` + Tauri events instead of SSE fetch. This ensures conversations are saved to the Tauri app's own DB and appear in the sidebar. If you add cloud provider features, update both the HTTP route handler (crates/llama-chat-web/src/routes/providers.rs) AND the Tauri command (src/main.rs `stream_provider`).

macOS-specific: Build with `npm run tauri:build:metal` (requires `brew install sccache`). The git submodule `deps/llama-cpp-rs` points to a Windows path (`E:/repo/llama-cpp-rs`) — on Mac, override with `git config submodule.deps/llama-cpp-rs.url https://github.com/utilityai/llama-cpp-rs.git` then `git clone https://github.com/utilityai/llama-cpp-rs.git deps/llama-cpp-rs && cd deps/llama-cpp-rs && git submodule update --init --recursive`. Native file picker (`rfd` crate) crashes in web server mode (no NSApplication) — `pick-file` uses `osascript` fallback on macOS, `pick-directory` reads from `models_directory` config instead. System info (`/api/system/usage`) uses `sysctl` for RAM/cores on macOS (unified memory reported as both RAM and VRAM). Perf timing: upstream llama-cpp-rs defaults `no_perf=true` (timing disabled) — `build_context_params()` sets it to `false` via unsafe FFI cast since the Rust binding doesn't expose it. Desktop-tools MCP server: `src/mcp_server.rs` builds as standalone binary `mcp_desktop_tools` — register with `claude mcp add desktop-tools -- ./target/release/mcp_desktop_tools`. Provides ~70 desktop automation tools (screenshot, click, OCR, window management) to Claude Code without needing a model loaded.

Cancel generation: Sets cancel flag → `execute_command_streaming()` kills subprocess within 200ms. Worker waits up to 3s for cancelled thread to finish before accepting new Generate command. `generating` field in ModelStatus tracks actual backend work state. Frontend shows friendly error messages instead of raw backend errors.

WebSocket /ws/status: Persistent health-check WebSocket with 20s ping keep-alive. Frontend ConnectionProvider (ConnectionContext.tsx) reconnects with exponential backoff (500ms → 5s). Red ConnectionBanner appears when server disconnects, disappears on reconnect. Skipped in Tauri mode.

System monitor: SystemResourcesContext polls at 10s background rate, switches to 3s when sidebar is open (setMonitorActive). Graphs render instantly on sidebar open because data is pre-populated. SystemUsage is a pure rendering consumer.

Engine scripts in `.claude/hooks/engines/run_homer.py` using faster-qwen3-tts (CUDA graphs, GPU-only). Venv at `E:/repo/faster-qwen3-tts/.venv/`. Available engines: "kitten" (KittenTTS CPU English), "homer" (faster-qwen3-tts GPU Spanish), "f5", "chatterbox", "cosyvoice".
